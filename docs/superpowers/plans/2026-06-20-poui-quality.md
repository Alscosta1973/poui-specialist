# PO-UI Quality — Auditoria Contínua de Qualidade

Varre todos os componentes Angular gerados pelo plugin e emite relatório de qualidade com 5 critérios. Nenhum arquivo é modificado.

## Uso

```
/poui-specialist:quality
```

Sem argumentos. Pode ser executado a qualquer momento — após geração, antes de commit, periodicamente.

---

## Passo 1 — Localizar raiz do projeto Angular

Verificar o diretório atual e subir até 3 níveis procurando `angular.json`:

```powershell
$angularRoot = $null
$dir = (Get-Location).Path
for ($i = 0; $i -lt 4; $i++) {
    if (Test-Path (Join-Path $dir "angular.json")) {
        $angularRoot = $dir
        break
    }
    $dir = Split-Path $dir -Parent
}
```

**Se não encontrado:** exibir e encerrar:

```
⚠ Nenhum projeto Angular encontrado — auditoria cancelada.
```

---

## Passo 2 — Coletar artefatos

**Componentes gerados pelo plugin:**

```powershell
$components = Get-ChildItem -Path (Join-Path $angularRoot "src/app") -Filter "*.component.ts" -Recurse |
    Where-Object { (Get-Content $_.FullName -Raw) -match '@generated  poui-specialist' }
```

**Se nenhum componente encontrado:** exibir e encerrar:

```
⚠ Nenhum componente gerado pelo plugin encontrado em src/app/.
```

**Arquivo de rotas:**

```powershell
$routesFile = Join-Path $angularRoot "src/app/app.routes.ts"
$routesContent = if (Test-Path $routesFile) { Get-Content $routesFile -Raw } else { '' }
```

---

## Passo 3 — Aplicar 5 regras de qualidade

Para cada arquivo em `$components`, ler o conteúdo e avaliar:

| # | Critério | Aprovado se | Severidade |
|---|---|---|---|
| 1 | OnPush | `ChangeDetectionStrategy\.OnPush` presente no arquivo | Crítico |
| 2 | Loading state | `finalize\(` presente no arquivo | Atenção |
| 3 | Error handling | `notification\.error\(` ou `catchError\(` presente no arquivo | Atenção |
| 4 | Cleanup observables | `takeUntilDestroyed` presente no arquivo | Atenção |

**Regra 5 — Lazy loading (apenas `app.routes.ts`):**

Para cada entrada com `path:` em `$routesContent`, verificar se a mesma entrada de rota contém `loadComponent:` (aprovado) ou `component:` (reprovado). Reportar como seção separada no relatório — não por componente individual.

---

## Passo 4 — Montar e exibir relatório

**Classificação por componente (critérios 1–4):**
- **Aprovado:** critérios 1, 2, 3 e 4 todos verdes
- **Atenção necessária:** qualquer critério 2, 3 ou 4 vermelho (critério 1 verde)
- **Crítico:** critério 1 (OnPush) vermelho, independente dos demais

**Exibir:**

```
## Relatório de Qualidade PO-UI
Auditado em: <data> | Componentes: N

### ✅ Aprovados (X)
- src/app/<módulo>/<entidade>/<arquivo>.component.ts — 4/4 critérios

### ⚠️ Atenção necessária (Y)
src/app/<módulo>/<entidade>/<arquivo>.component.ts — X/4 critérios

| Critério              | Status | Ação sugerida                                                            |
|-----------------------|--------|--------------------------------------------------------------------------|
| OnPush                | ✅     | —                                                                        |
| Loading state         | ❌     | Adicionar finalize(() => this.loading.set(false)) nas chamadas HTTP      |
| Error handling        | ✅     | —                                                                        |
| Cleanup observables   | ❌     | Adicionar .pipe(takeUntilDestroyed()) nos observables                    |

### 🔴 Críticos (Z)
<mesmo formato; OnPush aparece com ❌ e ação: Adicionar changeDetection: ChangeDetectionStrategy.OnPush no decorator @Component>

### Rotas auditadas (app.routes.ts)
| Rota                    | Lazy loading        |
|-------------------------|---------------------|
| <path-1>                | ✅ loadComponent    |
| <path-2>                | ❌ component direto |

---
Resumo: X aprovados · Y com atenção · Z críticos
Nenhuma alteração foi feita automaticamente. Edite os arquivos indicados para corrigir.
```

Se `$routesContent` estiver vazio (arquivo ausente): omitir seção "Rotas auditadas".
